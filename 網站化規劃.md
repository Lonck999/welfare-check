# welfare-check 網站化規劃

> 建立日期：2026-06-20
> 目標：把目前的 Claude Code Skill（welfare-check）改造成獨立網站，供他人查詢使用

---

## 一、背景與現況

目前 `welfare-check` 是一套 Claude Code Skill（見 [skills/welfare-check/SKILL.md](skills/welfare-check/SKILL.md)），運作方式：

- 靠 Claude Code 環境本身的多輪推理＋WebSearch/WebFetch 工具，逐題問答收集使用者資料
- 即時搜尋政府網站，比對 40 大類補助資格
- 輸出 HTML 報告

**這套運作方式無法直接搬到網站上**，因為它依賴 Claude Code 這個互動環境的 agent 能力，不是一支獨立程式。

---

## 二、核心困難點（排除金流）

### 1. 核心邏輯要整套重寫
[SKILL.md](skills/welfare-check/SKILL.md) 是寫給 Claude Code agent 看的「指示文件」，不是可獨立執行的程式。要做成網站，需要重建後端：自己管理對話狀態、自己呼叫 LLM API、自己處理工具呼叫迴圈與錯誤重試。

### 2. 即時網路搜尋的依賴最難複製
現在的核心價值在於即時查詢最新補助資訊。網站環境沒有 Claude Code 內建的搜尋工具，自接搜尋 API（Google/Bing）品質不穩定，容易抓到過期或非官方頁面。

### 3. 資料時效性的長期維運成本
補助項目（尤其地方政府加碼）會隨政策、年度預算不斷變動，需要長期人力或自動化機制持續追蹤更新，否則網站給出的資訊很快過期。

### 4. 回應時間與使用體驗
單次查詢若靠即時搜尋＋多次 LLM 推理，可能要數十秒到一兩分鐘，需設計成非同步流程（背景處理＋通知），不能讓使用者卡在頁面上等。

### 5. 併發與成本控管
多人同時查詢會疊加 API 呼叫量與費用，需要排隊機制、rate limiting，避免流量上升時成本失控或被惡意灌爆。

### 6. 個資蒐集與法律免責
使用者要填年齡、所得、財產、健康狀況等敏感資料，蒐集/儲存/傳輸都受個資法規範；資格判斷如有誤導致使用者錯失補助，需要明確免責聲明設計。

---

## 三、解決方案：資料庫化 + 規則比對引擎

**核心決定**：事先把所有福利、單位、所需文件等資訊存進資料庫，使用者填完資料後直接查資料庫比對資格，不在查詢當下呼叫 LLM 或即時搜尋。

### 優點
- 不再依賴 Claude Code 環境，邊際成本大幅下降（接近零），回應速度快
- 對「單次收費查詢」的商業模式更合理（成本可控）

### 帶來的新困難
1. **資格邏輯比想像中複雜**：排富條款、家庭人口計算方式、動產/不動產認定、特殊例外情況等，需要把政策文字轉換成結構化判斷規則（if-else／decision tree），且每次政策修改都要回頭改程式邏輯
2. **資料維護變成長期、持續的人力成本**（見第四節）
3. **涵蓋範圍可能反而變窄**：只能涵蓋事先建檔的項目，新政策或地方小眾加碼如果沒收錄會直接漏掉
4. **正確性責任更重**：資料庫資料過期或規則寫錯導致誤判，責任更明確地落在維護者身上

---

## 四、長期維運：每月批次更新流程

**策略**：用 Claude Code（之後可進一步自動化為 Claude API 腳本）每月跑一次 WebSearch/WebFetch，重新核對資料庫內容，而不是每次使用者查詢都即時搜尋。

### 1. 資料庫欄位要先加上「來源依據」
每筆福利資料除內容本身，需多存：
- `source_url`（官方來源網址）
- `source_excerpt`（從該網址擷取出來、支撐數值/門檻的原文片段）
- `last_verified_date`（最後確認日期）

### 2. 比對流程：只輸出「有來源佐證」的異動
每月批次針對資料庫內每一項福利：
- 重新搜尋＋WebFetch 該福利的官方頁面
- 從頁面擷取目前的金額/門檻/所需文件
- 跟資料庫現有值比對，**只有新舊不同時才列入異動清單**
- 異動清單格式：`福利名稱 | 欄位 | 舊值 | 新值 | 來源網址 | 原文引用片段`

### 3. 強制要求「網址必須真實可打開且內容對得上」
- AI 必須先用 WebFetch 實際抓到該網址內容，把比對所需的原文片段直接複製貼到異動清單裡（原文引用，不是 AI 自己改寫的描述）
- 如果 WebFetch 抓不到網址（404、改版、找不到對應內容），這筆異動標記「無法驗證」，**不能列入正式異動清單**，需人工另外處理（可能該福利已取消或改了網址）

### 4. 人工審核才能寫入資料庫
- 異動清單輸出成表格（Markdown / CSV），人工逐行核對：舊值 → 新值，點來源網址確認
- 只有「核准」的異動才寫入資料庫；未核准的不動，下個月再檢查一次
- **AI 的角色是「找差異 + 附證據」，寫入資料庫的決定權永遠在人工**

### 5. 新福利／疑似下架的福利要分開處理
- AI 搜尋到資料庫沒有的新福利 → 另列「新增候選清單」，一樣要附來源網址
- 資料庫裡的福利，AI 搜尋不到對應頁面 → 標記「疑似已下架」，**不自動刪除**，等人工確認

### 6. 版本紀錄
每次更新保留 diff/log，記錄誰核准了什麼異動、何時生效，方便回溯與除錯。

### 7. 對使用者的透明標示
網站需明確標示「資料更新日期：YYYY/MM/DD」，並在免責聲明中說明「請以各單位最新公告為準」，劃清時效落差的責任界線。

## 五、待辦：自動化批次更新（暫緩，排在後面做）

目前構想是先用 Claude Code 手動跑每月更新，長期則需要把這個流程寫成腳本，用 Claude API（+ web search tool）排程自動執行，不依賴人工每月記得操作。

**架構待規劃項目（之後再展開）：**
- Claude API 呼叫腳本（含 web search tool 串接）
- 排程機制（cron / 雲端排程服務）
- 異動清單產出格式與審核介面
- 審核通過後寫入資料庫的流程

---

## 六、暫不考慮：金流整合

商業模式設想是單次查詢收費 10 元，但金流整合（綠界/藍新/TapPay 等多要求公司登記、且需評估 API 成本是否高於 10 元）**先不在本次規劃範圍內**，待核心查詢功能與資料維護機制穩定後再評估。

## 七、整體步驟總覽

**前提確認**：全端自己寫程式開發；使用者不需註冊帳號，每次查詢都是一次性填表即查即丟（不保留歷史記錄，最簡單）。

### 階段 1：技術選型與基礎建設

**1.1 技術棧選型（已決定，見對應 Issue）**
- [x] 前端框架：**Vue 3 + Vite**（先不用 Nuxt，之後視需要再整合，取得 SSR/檔案路由等全端功能）— [#3](https://github.com/Lonck999/welfare-check/issues/3)
- [x] 後端框架/語言：**Node.js + Fastify**（與前端同語言、學習曲線低，效能優於 Express）— [#4](https://github.com/Lonck999/welfare-check/issues/4)
- [x] 資料庫：**Neon（Serverless PostgreSQL，代管服務）**（不需要 Supabase 內建 Auth，因確定不做帳號系統）— [#5](https://github.com/Lonck999/welfare-check/issues/5)
- [x] ORM：**Drizzle**（TypeScript 原生、輕量，與 Neon 搭配為社群常見組合）— [#6](https://github.com/Lonck999/welfare-check/issues/6)

**1.2 網域與主機（已決定，見對應 Issue）**
- [x] 網域：**先用 Railway 免費子網域**（如 `xxx.up.railway.app`），待有實際需求再購買正式網域 — [#7](https://github.com/Lonck999/welfare-check/issues/7)
- [x] 主機/雲端代管平台：**Railway**（前後端都部署在此，對 Node.js 支援度好）— [#8](https://github.com/Lonck999/welfare-check/issues/8)
- [x] SSL：**使用 Railway 免費子網域自動配發的 SSL**，免額外設定；之後綁定正式網域時 Railway 也支援自動申請憑證 — [#9](https://github.com/Lonck999/welfare-check/issues/9)
- [x] 流量/併發上限：**先用免費方案測試**，上線後依第 1.4 節監控工具觀察實際用量，再評估是否升級付費方案 — [#10](https://github.com/Lonck999/welfare-check/issues/10)

**1.3 環境與部署流程**
- [ ] 建立 Git repository，規劃分支策略（如 main / dev）
- [ ] 建立本地開發環境，可連線測試用資料庫

- [ ] 設定環境變數管理（資料庫連線字串、API 金鑰等，不寫進程式碼／不進版控）
